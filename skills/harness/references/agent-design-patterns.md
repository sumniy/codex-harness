# Agent Design Patterns (Codex CLI)

## 실행 모드: 서브 에이전트 기반 아키텍처

Codex CLI는 **서브 에이전트 모델**을 사용한다. 메인 에이전트가 서브 에이전트를 생성하고, 서브 에이전트는 작업 결과를 메인에게만 반환한다. 에이전트 팀(팀원 간 직접 통신)은 지원하지 않지만, 아래에서 설명하는 모든 아키텍처 패턴은 서브 에이전트 조합과 **파일 기반 데이터 전달**로 동일하게 구현할 수 있다.

### 서브 에이전트 (Sub-agents) — 유일한 실행 모드

메인 에이전트가 서브 에이전트를 생성한다. 서브 에이전트는 작업 결과를 메인에게만 반환하고 서로 통신하지 않는다.

```
[메인] → [서브A] → 결과 반환
      → [서브B] → 결과 반환
      → [서브C] → 결과 반환
```

**핵심 메커니즘:**
- 서브 에이전트 생성: 메인이 프롬프트와 `subagent_type`을 지정하여 스폰
- 파일 기반 데이터 전달: 서브 에이전트 간 데이터 교환은 `_workspace/` 디렉토리의 파일을 통해 수행
- 백그라운드 실행: `run_in_background`로 병렬 실행 가능

**특징:**
- 가볍고 빠름
- 결과가 메인 컨텍스트로 요약 반환
- 토큰 효율적

**제약:**
- 서브 에이전트 간 직접 통신 불가
- 메인이 모든 조율 담당
- 실시간 협업/도전 불가

### 에이전트 간 데이터 교환: 파일 기반 전달

서브 에이전트 간 직접 통신이 불가능하므로, **파일 기반 데이터 전달**이 유일한 에이전트 간 데이터 교환 방법이다.

```
[서브A] → _workspace/analysis-result.json → [메인이 읽고 서브B에 전달]
[서브B] → _workspace/design-spec.md → [메인이 읽고 서브C에 전달]
```

**핵심 원칙:**
- 모든 중간 산출물은 `_workspace/` 디렉토리에 파일로 저장
- 메인 에이전트가 파일을 읽어서 다음 서브 에이전트의 프롬프트에 포함하거나, 파일 경로를 전달
- 구조화된 데이터는 JSON, 문서는 Markdown 형식 권장
- 파일명에 단계/순서를 포함하여 추적 가능하게 관리 (예: `01-analysis.json`, `02-design.md`)

### 모드 선택 의사결정 트리

```
작업이 단일 서브 에이전트로 완료 가능한가?
├── Yes → 서브 에이전트 1개 생성
│
└── No → 작업을 독립 단위로 분해 가능한가?
          ├── Yes → 병렬 서브 에이전트 (팬아웃)
          │         각 서브 에이전트가 독립 작업 수행 → 메인이 결과 통합
          │
          └── No → 순차 서브 에이전트 (파이프라인)
                    이전 서브 에이전트의 산출물을 파일로 저장 → 다음 서브 에이전트가 참조
```

> **핵심 원칙:** 서브 에이전트 간 통신이 필요한 경우, 메인 에이전트가 중개자 역할을 하거나 파일 기반 데이터 전달을 사용한다. 패턴의 선택은 작업의 독립성과 순차 의존성에 따라 결정한다.

## 서브 에이전트 기반 아키텍처 유형

### 1. 파이프라인 (Pipeline)
순차적 작업 흐름. 이전 에이전트의 출력이 다음 에이전트의 입력.

```
[분석] → _workspace/분석결과.json → [설계] → _workspace/설계서.md → [구현] → _workspace/코드 → [검증]
```

**적합한 경우:** 각 단계가 이전 단계의 산출물에 강하게 의존
**예시:** 소설 집필 — 세계관 → 캐릭터 → 플롯 → 집필 → 편집
**주의:** 병목이 전체 파이프라인을 지연시킴. 각 단계를 가능한 독립적으로 설계할 것.
**구현:** 메인 에이전트가 순차적으로 서브 에이전트를 생성하고, 각 단계의 산출물을 `_workspace/`에 파일로 저장하여 다음 단계에 전달.

### 2. 팬아웃/팬인 (Fan-out/Fan-in)
병렬 처리 후 결과 통합. 독립적 작업을 동시 수행.

```
         ┌→ [전문가A] → _workspace/resultA.json ─┐
[분배] → ├→ [전문가B] → _workspace/resultB.json ─┼→ [메인이 통합]
         └→ [전문가C] → _workspace/resultC.json ─┘
```

**적합한 경우:** 동일 입력에 대해 서로 다른 관점/영역의 분석이 필요
**예시:** 종합 리서치 — 공식/미디어/커뮤니티/배경 동시 조사 → 통합 보고
**주의:** 통합 단계의 품질이 전체 품질을 결정.
**구현:** `run_in_background`로 서브 에이전트를 병렬 생성. 각 서브 에이전트는 결과를 `_workspace/`에 파일로 저장. 메인이 모든 결과 파일을 읽어 통합 서브 에이전트에 전달하거나 직접 통합.

### 3. 전문가 풀 (Expert Pool)
상황에 따라 적절한 전문가를 선택 호출.

```
[라우터] → { 전문가A | 전문가B | 전문가C }
```

**적합한 경우:** 입력 유형에 따라 다른 처리가 필요
**예시:** 코드 리뷰 — 보안/성능/아키텍처 전문가 중 해당 영역만 호출
**주의:** 라우터의 분류 정확도가 핵심.
**구현:** 메인 에이전트가 라우터 역할을 하여 입력을 분석하고, 적합한 `subagent_type`의 서브 에이전트만 호출.

### 4. 생성-검증 (Producer-Reviewer)
생성 에이전트와 검증 에이전트가 쌍으로 동작.

```
[생성] → _workspace/output.md → [검증] → (문제시) → [생성] 재실행
```

**적합한 경우:** 산출물의 품질 보장이 중요하고 객관적 검증 기준이 존재
**예시:** 웹툰 — artist 생성 → reviewer 검수 → 문제 패널 재생성
**주의:** 무한 루프 방지를 위해 최대 재시도 횟수(2~3회) 설정 필수.
**구현:** 메인 에이전트가 생성 서브 에이전트 → 검증 서브 에이전트를 순차 호출. 검증 결과에 따라 메인이 재생성 여부를 결정하고, 검증 피드백을 파일로 저장하여 재생성 시 참조.

### 5. 감독자 (Supervisor)
중앙 에이전트가 작업 상태를 관리하며 하위 에이전트에 동적으로 작업을 분배.

```
         ┌→ [워커A]
[감독자] ─┼→ [워커B]    ← 감독자(메인)가 상태를 보고 동적 분배
         └→ [워커C]
```

**적합한 경우:** 작업량이 가변적이거나 런타임에 작업 분배를 결정해야 할 때
**예시:** 대규모 코드 마이그레이션 — 감독자가 파일 목록을 분석하고 워커들에게 배치 할당
**팬아웃과의 차이:** 팬아웃은 사전에 작업을 고정 분배, 감독자는 진행 상황을 보며 동적 조정
**주의:** 감독자가 병목이 되지 않도록 위임 단위를 충분히 크게 설정.
**구현:** 메인 에이전트가 감독자 역할. `_workspace/task-queue.json`에 작업 목록을 관리하고, 워커 서브 에이전트를 병렬 생성. 각 워커의 완료 결과를 확인한 후 다음 배치를 할당.

### 6. 계층적 위임 (Hierarchical Delegation)
상위 에이전트가 하위 에이전트에 재귀적으로 위임. 복잡한 문제를 단계적으로 분해.

```
[총괄] → [팀장A] → [실무자A1]
                  → [실무자A2]
       → [팀장B] → [실무자B1]
```

**적합한 경우:** 문제가 자연스럽게 계층적으로 분해되는 구조
**예시:** 풀스택 앱 개발 — 총괄 → 프론트엔드팀장 → (UI/로직/테스트) + 백엔드팀장 → (API/DB/테스트)
**주의:** 깊이 3단계 이상은 지연과 컨텍스트 손실이 커짐. 2단계 이내 권장.
**구현:** 서브 에이전트가 자신의 서브 에이전트를 생성하는 재귀 구조. 각 단계의 산출물은 `_workspace/`에 파일로 저장하여 상위 에이전트가 접근 가능.

## 복합 패턴

실전에서는 단일 패턴보다 복합 패턴이 흔하다:

| 복합 패턴 | 구성 | 예시 |
|----------|------|------|
| **팬아웃 + 생성-검증** | 병렬 생성 후 각각 검증 | 다국어 번역 — 4개 언어 병렬 번역 → 각각 네이티브 리뷰어 검수 |
| **파이프라인 + 팬아웃** | 순차 단계 중 일부를 병렬화 | 분석(순차) → 구현(병렬) → 통합 테스트(순차) |
| **감독자 + 전문가 풀** | 감독자가 전문가를 동적 호출 | 고객 문의 처리 — 감독자가 문의 분류 후 적합한 전문가 할당 |

### 복합 패턴에서의 구현 전략

**메인 에이전트가 오케스트레이터 역할을 하며, 파일 기반 데이터 전달로 서브 에이전트 간 정보를 중개한다.**

| 시나리오 | 구현 전략 | 파일 전달 방식 |
|---------|----------|--------------|
| **리서치 + 분석** | 병렬 서브 에이전트 → 메인이 결과 통합 후 분석 서브 에이전트에 전달 | `_workspace/research-*.json` → `_workspace/integrated-analysis.md` |
| **설계 + 구현 + 검증** | 순차 파이프라인, 각 단계 산출물 파일 저장 | `_workspace/design.md` → `_workspace/impl/` → `_workspace/verification.json` |
| **감독자 + 워커** | 메인이 작업 큐 파일 관리, 워커 병렬 실행 | `_workspace/task-queue.json`, `_workspace/results/worker-*.json` |
| **생성 + 검증** | 순차 호출 + 피드백 파일로 반복 | `_workspace/draft.md` → `_workspace/review-feedback.json` → 재생성 |

> 서브 에이전트 간 실시간 통신은 불가하므로, 메인 에이전트가 중간 결과를 읽고 다음 서브 에이전트에 전달하는 **허브-스포크 모델**을 사용한다.

## 에이전트 타입 선택

서브 에이전트를 호출할 때 `subagent_type` 파라미터로 타입을 지정한다.

### 빌트인 타입

| 타입 | 도구 접근 | 적합한 용도 |
|------|----------|-----------|
| `default` | 전체 (WebSearch, WebFetch 포함) | 웹 조사, 범용 작업 |
| `explorer` | 읽기 전용 (Edit/Write 없음) | 코드베이스 탐색, 분석 |
| `Plan` | 읽기 전용 (Edit/Write 없음) | 아키텍처 설계, 계획 수립 |

### 커스텀 타입

`.codex/agents/{name}.toml`에 에이전트를 정의하면 `subagent_type: "{name}"`으로 호출할 수 있다. 커스텀 에이전트는 전체 도구에 접근 가능.

### 선택 기준

| 상황 | 권장 | 이유 |
|------|------|------|
| 역할이 복잡하고 여러 세션에서 재사용 | **커스텀 타입** (`.codex/agents/`) | 페르소나와 작업 원칙을 파일로 관리 |
| 단순 조사/수집이고 프롬프트만으로 충분 | **`default`** + 상세 프롬프트 | 에이전트 파일 불필요, 프롬프트에 지시 포함 |
| 코드 읽기만 필요 (분석/리뷰) | **`explorer`** | 실수로 파일 수정하는 것을 방지 |
| 설계/계획만 필요 | **`Plan`** | 분석에 집중, 코드 변경 방지 |
| 파일 수정이 필요한 구현 작업 | **커스텀 타입** | 전체 도구 접근 + 전문 지시 |

**원칙:** 모든 에이전트는 반드시 `.codex/agents/{name}.toml` 파일로 정의한다. 빌트인 타입이라도 에이전트 정의 파일을 생성하여 역할·원칙·프로토콜을 명시한다. 파일로 존재해야 다음 세션에서 재사용 가능하고, 데이터 전달 프로토콜이 명시되어야 협업 품질이 보장된다.

**모델:** 모든 에이전트는 최상위 모델 (o3, gpt-5.4 등)을 사용한다. 서브 에이전트 호출 시 반드시 최상위 모델을 명시한다.

## 에이전트 정의 구조

```toml
[agent]
name = "agent-name"
description = "1-2문장 역할 설명. 트리거 키워드 나열."

[agent.role]
summary = "역할 한줄 요약"
persona = "당신은 [도메인]의 [역할] 전문가입니다."

[agent.core_roles]
roles = [
  "역할1",
  "역할2",
]

[agent.principles]
rules = [
  "원칙1",
  "원칙2",
]

[agent.io_protocol]
input = "[어디서 무엇을 받는지]"
output = "[어디에 무엇을 쓰는지]"
format = "[파일 포맷, 구조]"

[agent.data_passing]
reads_from = ["_workspace/이전-단계-산출물.json"]
writes_to = ["_workspace/현재-단계-산출물.json"]
file_format = "JSON 또는 Markdown"

[agent.error_handling]
on_failure = "[실패 시 행동]"
on_timeout = "[타임아웃 시 행동]"

[agent.collaboration]
dependencies = ["다른 에이전트와의 관계"]
```

## 에이전트 분리 기준

| 기준 | 분리 | 통합 |
|------|------|------|
| 전문성 | 영역이 다르면 분리 | 영역이 겹치면 통합 |
| 병렬성 | 독립 실행 가능하면 분리 | 순차 종속이면 통합 고려 |
| 컨텍스트 | 컨텍스트 부담이 크면 분리 | 가볍고 빠르면 통합 |
| 재사용성 | 다른 워크플로우에서도 쓰면 분리 | 이 워크플로우에서만 쓰면 통합 고려 |

## 스킬 vs 에이전트 구분

| 구분 | 스킬 (Skill) | 에이전트 (Agent) |
|------|-------------|-----------------|
| 정의 | 절차적 지식 + 도구 번들 | 전문가 페르소나 + 행동 원칙 |
| 위치 | `.agents/skills/` | `.codex/agents/` |
| 트리거 | `$skill-name`으로 호출 | 서브 에이전트로 명시적 호출 |
| 크기 | 작은~큰 (워크플로우) | 작은 (역할 정의) |
| 용도 | "어떻게 하는가" | "누가 하는가" |

스킬은 에이전트가 작업을 수행할 때 참조하는 **절차적 가이드**.
에이전트는 스킬을 활용하는 **전문가 역할 정의**.

## 스킬 ↔ 에이전트 연결 방식

에이전트가 스킬을 활용하는 3가지 방식:

| 방식 | 구현 | 적합한 경우 |
|------|------|-----------|
| **`$skill-name` 호출** | 에이전트 프롬프트에 `$skill-name 호출` 명시 | 스킬이 독립 워크플로우이고 명시적 호출 가능한 경우 |
| **프롬프트 내 인라인** | 에이전트 정의 내에 스킬 내용을 직접 포함 | 스킬이 짧고(50줄 이하) 이 에이전트 전용인 경우 |
| **레퍼런스 로드** | `Read`로 스킬의 references/ 파일을 필요 시 로드 | 스킬 내용이 크고 조건부로만 필요한 경우 |

권장: 재사용성이 높으면 `$skill-name` 호출, 전용이면 인라인, 대용량이면 레퍼런스 로드.
